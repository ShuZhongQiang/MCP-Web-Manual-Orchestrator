# 任务描述示例

本文件提供一套适合 `MCP Web Manual Orchestrator` 的标准任务描述写法，分为两种模式：

- 精准描述模式：适合必须稳定复现、字段值固定、验收标准明确的任务。
- 大概描述模式：适合只给业务目标，让 Agent 在既定边界内自行补齐细节的任务。

这两种模式都比“帮我点一下页面并生成手册”这类模糊表述更稳定。

## 先说结论

如果你希望运行结果更稳定，任务描述里至少要明确以下 6 类信息：

1. 起始页面或入口 URL
2. 业务目标
3. 关键路径或页面名称
4. 需要填写的数据范围
5. 成功判定信号
6. 最终产物要求

## 模式选择建议

### 什么时候用精准描述模式

- 你要生成可复现的 SOP、培训手册、交付材料
- 你希望字段值固定，方便后续核对截图
- 你希望模型不要自行发挥业务值

### 什么时候用大概描述模式

- 你只关心完成业务目标，不要求每个字段值都固定
- 你允许 Agent 按默认字段策略自动补齐
- 你希望先提高任务完成率，再逐步收紧约束

## 精准描述模式

### 推荐模板

```md
任务名称：
<一句话描述任务目标>

系统与入口：
- 系统名称：<系统名>
- 起始 URL：<URL>
- 若已登录可直接从当前页面继续，否则先完成登录

业务目标：
- <明确要完成的业务动作>

必须执行的关键路径：
1. <页面或菜单路径 1>
2. <页面或菜单路径 2>
3. <页面或菜单路径 3>

表单填写要求：
- <字段名 1>：<固定值>
- <字段名 2>：<固定值>
- <字段名 3>：<固定值>
- 若出现未明确字段：
  - 必填字段：按页面语义补齐
  - 非必填字段：可跳过，除非影响提交

选项与枚举要求：
- <字段名> 优先选择：<选项值>
- 若目标选项不存在：记录原因并停止瞎选

成功判定：
- 至少满足以下之一：
  - <成功 toast / 提示文案>
  - <弹窗关闭>
  - <列表中出现新记录>
  - <URL 或页面状态变化>

截图与手册要求：
- 对每个关键操作截图
- 最终生成 HTML 手册
- 最后一张图必须体现业务成功结果

执行约束：
- 只能使用 MCP 工具，不要生成 Playwright/JS 脚本
- 先用 execution records 驱动手册，不要凭空补步骤
- 若进入表单场景，先识别 active layer / form / validation，再填写和提交
```

### 精准描述示例

```md
任务名称：
生成“咖啡后台管理系统-新增会员”操作手册

系统与入口：
- 系统名称：咖啡后台管理系统
- 起始 URL：http://localhost:5173/login
- 若页面已经处于登录后状态，可直接继续，不必重复登录

业务目标：
- 新增 1 个测试会员，并生成带截图的 HTML 操作手册

必须执行的关键路径：
1. 进入登录页并点击“登录”
2. 点击左侧“会员管理”
3. 点击“添加会员”
4. 完成表单填写并提交
5. 返回会员列表并确认新增结果

表单填写要求：
- 姓名：张三
- 手机号：13800138000
- 邮箱：zhangsan@example.com
- 等级：普通会员
- 积分：0
- 若还有其他必填字段，按页面提示补齐
- 若字段非必填且不影响提交，可以跳过

选项与枚举要求：
- 等级字段必须优先选择“普通会员”
- 若“普通会员”不存在，不要改选其他等级，直接记录失败原因

成功判定：
- 至少满足以下之一：
  - 出现成功提示
  - 添加会员弹窗关闭
  - 会员列表中可见“张三”或手机号“13800138000”

截图与手册要求：
- 登录、进入会员管理、打开新增弹窗、填写关键字段、提交、列表确认，这些步骤都要截图
- 最终生成 HTML 手册
- 最后一张截图优先展示会员列表中的新增结果

执行约束：
- 只能使用 MCP 工具，不要额外写脚本
- 对同一个逻辑步骤，保持一致的 step_id，不要把多个逻辑步骤混用成同一个 step
- generate_manual 优先依赖 execution records；steps_json 只补标题、摘要、模块信息
- 不要在 steps_json 中写入未实际执行的步骤
- 若进入表单场景，先调用 inspect_active_layer、inspect_form、compile_form_plan、inspect_validation，再驱动填写和提交
```

### 适用特点

- 最稳定
- 最适合生成交付件
- 最适合做回归对比

## 大概描述模式

### 推荐模板

```md
任务目标：
<一句话说明业务目标>

已知条件：
- 系统名称：<系统名>
- 入口：<URL 或当前页面>
- 当前环境：<测试 / 演示 / 本地>

希望 Agent 自主完成的部分：
- 自行识别页面路径
- 自行补齐合理的测试数据
- 自行判断哪些截图是关键步骤

边界要求：
- 只能在 <某模块 / 某菜单> 范围内操作
- 不要删除数据
- 不要修改已有重要记录
- 若需要补齐表单，优先使用安全的测试数据

成功标准：
- <描述业务结果>
- 最终输出 HTML 手册

额外说明：
- 若页面文案和我描述不完全一致，以实际页面语义接近项为准
- 若存在多个可行路径，选择最短且最稳定的一条
```

### 大概描述示例

```md
任务目标：
帮我在咖啡后台管理系统里新增一个测试会员，并输出操作手册。

已知条件：
- 系统名称：咖啡后台管理系统
- 入口：http://localhost:5173/login
- 当前环境：本地演示环境

希望 Agent 自主完成的部分：
- 自行找到登录后进入“会员管理”的路径
- 自行识别新增会员表单里的必填项
- 自行补齐合理的测试数据
- 自行选择关键截图，最后生成 HTML 手册

边界要求：
- 只允许操作会员管理相关页面
- 不要删除已有会员
- 新增的数据必须是明显的测试数据
- 如果表单里有下拉框，优先选择非占位项中的默认安全选项

成功标准：
- 会员新增成功
- 最终能看到新增后的会员列表或成功结果
- 输出 HTML 手册

额外说明：
- 如果页面已有登录态，可以直接继续
- 如果系统提示字段校验失败，先补齐字段再重试
- 若找不到明确的“保存”按钮，可以检查弹窗底部是否存在“确定”或同义按钮
```

### 适用特点

- 更省提示词
- 更适合探索型任务
- 对模型编排能力要求更高

## 推荐附加约束

无论你使用哪种模式，建议在任务描述末尾追加以下约束段：

```md
编排要求：
- 所有操作必须通过 MCP 工具完成
- 每个逻辑步骤开始前优先调用 begin_step
- 对同一个逻辑步骤，navigate / click / input_text / highlight_and_capture 要共享同一个 step_id
- 进入表单场景后，先 inspect_active_layer，再 inspect_form，再 compile_form_plan，再 inspect_validation
- generate_manual 以 execution records 为主，steps_json 只补充元信息
- 若无法确认业务成功，不要把“点击成功”当成“业务成功”
```

## 不推荐的任务描述

下面这些写法过于模糊，容易导致结果漂移：

```md
帮我操作一下这个系统并生成手册。
```

```md
去新增一个会员，自己看着办。
```

```md
打开网页，把该填的都填了，然后截图。
```

问题不在于模型“笨”，而在于约束不完整：

- 没有入口
- 没有成功判定
- 没有字段边界
- 没有产物要求
- 没有限制自由发挥范围

## 实战建议

1. 首次跑通时，优先使用精准描述模式。
2. 当流程已稳定后，再收缩成大概描述模式，验证系统的泛化能力。
3. 如果你要比较模型差异，任务描述中必须固定字段值、成功信号、输出要求。
4. 如果你要提高完成率，先固定“成功判定”和“禁止自由发挥的边界”，再优化其他描述。
